# External Attendance API — Design Spec

**Date:** 2026-05-05
**Status:** Draft
**Author:** AI assistant (brainstorming session with user)

## Overview

Add a cross-origin (CORS-enabled) external API for attendance that allows other websites to:
1. Perform **auto check-in** (registers check-in + auto checkout after 8 hours, no separate checkout needed)
2. **Read all attendance data**

API key authentication replaces session cookies (which don't work cross-origin). Admin manages API keys via a settings UI.

Primary use case: QR code-based attendance where the QR is generated by a separate web application that calls this API.

## Architecture

Dedicated namespace `app/api/external/` with its own middleware for CORS + API key auth. Reuses internal service functions where possible.

```
app/api/external/
  utils.ts              # Shared: validateApiKey(), setCorsHeaders(), externalResponse()
  attendance/
    route.ts            # GET /api/external/attendance
    services.ts         # getAttendanceData
    auto-checkin/
      route.ts          # POST /api/external/attendance/auto-checkin
      services.ts       # validateLocationData, autoCheckIn

middleware.ts (root)    # Extended: CORS headers for /api/external/:path*, OPTIONS preflight
```

> **Note:** Next.js App Router `middleware.ts` only exists at project root. CORS headers are set there (Edge-compatible, no DB access). API key validation (requires Prisma/Node.js) happens via `validateApiKey()` helper called at the top of each route handler. The root `middleware.ts` handles OPTIONS preflight to avoid hitting route handlers for preflight requests.

### Request Flow

```
External website → OPTIONS preflight → root middleware (CORS headers) → 204
External website → POST /api/external/attendance/auto-checkin
  → root middleware (CORS headers)
  → route handler → validateApiKey() → services (validate, process, log) → JSON response
```

CORS is handled in root middleware (Edge-compatible, no DB access). API key validation happens in the route handler via shared `validateApiKey()` helper (Node.js runtime, needs Prisma).

## CORS Configuration

Root `middleware.ts` sets CORS headers for all `/api/external/*` routes:

| Header | Value |
|---|---|
| `Access-Control-Allow-Origin` | `*` (or whitelist from DB Setting) |
| `Access-Control-Allow-Methods` | `GET, POST, OPTIONS` |
| `Access-Control-Allow-Headers` | `Content-Type, x-api-key` |
| `Access-Control-Max-Age` | `86400` |

`OPTIONS` preflight is handled in root middleware — returns `204 No Content` without hitting route handlers. CORS is Edge-only (no DB access), API key auth is deferred to route handlers.

## Database

### New Model: `ApiKey`

```prisma
model ApiKey {
  id          String    @id @default(uuid())
  key         String    @unique       // bcryptjs hash
  prefix      String                  // first 8 chars of raw key (plaintext, for UI)
  name        String                  // label e.g. "QR Scanner App"
  scope       String    @default("attendance:readwrite")
  isActive    Boolean   @default(true)
  createdBy   String    @map("created_by")
  lastUsedAt  DateTime? @map("last_used_at")
  createdAt   DateTime  @default(now()) @map("created_at")
  updatedAt   DateTime  @updatedAt @map("updated_at")

  creator User @relation(fields: [createdBy], references: [id])

  @@map("api_keys")
}
```

**Key format:** `api_live_` + 48 random alphanumeric chars. Stored in DB as bcryptjs hash (same pattern as user passwords). Only shown to admin once at creation time.

**Scope values:**

| Scope | Permissions |
|---|---|
| `attendance:readwrite` | All: read + auto-checkin |
| `attendance:read` | Read-only |
| `attendance:auto-checkin` | Auto check-in only |

## Endpoints

### 1. `POST /api/external/attendance/auto-checkin`

Register an automatic check-in with 8-hour work duration.

**Auth:** `x-api-key` header, scope `attendance:auto-checkin` or `attendance:readwrite`

**Request Body:**
```json
{
  "userId": "string (required)",
  "latitude": -6.2088,
  "longitude": 106.8456,
  "accuracy": 12.5,
  "notes": "via QR scan (optional)"
}
```

**Validation:**
- `userId` must reference an existing, active user
- `latitude` / `longitude` required, valid float ranges

**Processing:**
1. Validate API key scope
2. Validate user exists and is active
3. Check for existing record today (`@@unique([userId, date])`) — reject if already exists
4. Reverse geocode coordinates via `lib/location.ts`
5. Determine `status` via existing `determineAttendanceStatus()` logic
6. Create `AbsensiRecord`:
   - `checkInTime = now()`
   - `checkOutTime = now() + 8 hours`
   - `workHours = 8.00`
   - `overtimeHours = 0.00`
   - All check-in GPS fields populated
   - `notes` from request body
7. Log activity via `prisma.activityLog.create()` with `action: "EXTERNAL_API_AUTO_CHECKIN"`

**Response 201:**
```json
{
  "success": true,
  "data": {
    "id": "...",
    "checkInTime": "2026-05-05T08:00:00.000Z",
    "checkOutTime": "2026-05-05T16:00:00.000Z",
    "workHours": "8.00",
    "status": "present",
    "checkInAddress": "Jl. Sudirman, Jakarta"
  }
}
```

**Errors:**
| Status | Message |
|---|---|
| 401 | Invalid or missing API key |
| 403 | API key lacks required scope |
| 400 | Invalid body / missing fields |
| 404 | User not found or inactive |
| 409 | Attendance already exists for today |

### 2. `GET /api/external/attendance`

Fetch attendance records.

**Auth:** `x-api-key` header, scope `attendance:read` or `attendance:readwrite`

**Query Parameters:**

| Param | Type | Default | Description |
|---|---|---|---|
| `date` | ISO date | today | Single date filter |
| `dateFrom` | ISO date | — | Start of date range |
| `dateTo` | ISO date | — | End of date range |
| `userId` | string | — | Filter by user |
| `limit` | number | 50 | Max 200 |
| `offset` | number | 0 | Pagination offset |

If `date` is provided, `dateFrom`/`dateTo` are ignored.

**Response 200:**
```json
{
  "success": true,
  "data": [
    {
      "id": "...",
      "userId": "...",
      "userName": "John Doe",
      "date": "2026-05-05",
      "checkInTime": "2026-05-05T08:00:00.000Z",
      "checkOutTime": "2026-05-05T16:00:00.000Z",
      "checkInLatitude": "-6.20880000",
      "checkInLongitude": "106.84560000",
      "checkInAddress": "Jl. Sudirman, Jakarta",
      "checkOutLatitude": null,
      "checkOutLongitude": null,
      "checkOutAddress": null,
      "workHours": "8.00",
      "overtimeHours": "0.00",
      "lateMinutes": 0,
      "status": "present"
    }
  ],
  "pagination": {
    "total": 150,
    "limit": 50,
    "offset": 0
  }
}
```

**Errors:**
| Status | Message |
|---|---|
| 401 | Invalid or missing API key |
| 403 | API key lacks required scope |

## API Key Management (UI)

### Route: `/settings/api-keys` (admin only)

Page with:
- **Table**: prefix, name, scope, last used timestamp, active status, create date
- **Generate button** → modal with `name` (text input) + `scope` (dropdown) → creates key, shows full raw key in modal with copy button and warning "Save this key — it won't be shown again"
- **Revoke**: toggle button per row → sets `isActive = false`
- Delete: soft-delete (same as revoke)

### API endpoints for management (internal, session auth):

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/settings/api-keys` | List all keys (prefix only, never reveal full key) |
| `POST` | `/api/settings/api-keys` | Generate new key |
| `PUT` | `/api/settings/api-keys/[id]` | Update name, toggle active |
| `DELETE` | `/api/settings/api-keys/[id]` | Soft delete (set isActive false) |

## Error Handling

All external API responses use consistent envelope:

```json
{ "success": true, "data": ... }
{ "success": false, "error": "Human-readable message" }
```

HTTP status codes follow REST conventions (200, 201, 400, 401, 403, 404, 409, 500).

## Activity Logging

All external API requests log to `ActivityLog` with:
- `action`: `EXTERNAL_API_AUTO_CHECKIN`, `EXTERNAL_API_READ_ATTENDANCE`
- `userId`: from the API key context (admin who created the key, or a system user)
- `details`: JSON with `apiKeyId`, `prefix`, `endpoint`, `ip`

## Testing

- **Unit tests** for service functions (`autoCheckIn`, `getAttendanceData`, `validateApiKey`) in `*.services.test.ts` (follow existing pattern)
- **Integration tests** that hit endpoints with `x-api-key` header (optional, can be manual via curl/Postman)
